## User Input

```text
$ARGUMENTS
```

입력이 비어있지 않다면 반드시 고려한 뒤 진행합니다.

## 한국어 작성 규칙

- 사용자에게 보여주는 질문/요약/보고는 한국어로 작성합니다.
- 명령어, 경로, 파일명, 코드 블록, ID 토큰(`CHK001`, `[US1]`)은 유지합니다.

## 핵심 개념

체크리스트는 **요구사항 문서의 품질을 검증하는 단위 테스트**입니다.

- 구현 동작 테스트를 만들지 않습니다.
- 코드가 맞는지 확인하지 않습니다.
- 요구사항의 완전성/명확성/일관성/측정가능성을 점검합니다.

## Execution Steps

1. **Setup**
   - `.specify/scripts/bash/check-prerequisites.sh --json` 실행
   - `FEATURE_DIR`, `AVAILABLE_DOCS` 파싱
2. **의도 명확화**
   - 사용자 입력 + 문서 시그널을 기반으로 최대 3개 질문(Q1~Q3) 생성
   - 범위/위험/깊이/대상 독자/제외 범위를 우선 질문
   - 이미 명확하면 질문 생략
   - 필요 시 Q4/Q5 추가(최대 5개)
3. **컨텍스트 로드**
   - `spec.md`(필수)
   - `plan.md`, `tasks.md`(있으면 로드)
   - 활성 포커스와 관련된 최소 구간만 읽기
4. **체크리스트 생성**
   - `FEATURE_DIR/checklists/` 디렉터리 생성(없으면)
   - 파일명: `[domain].md` (예: `ux.md`, `api.md`, `security.md`)
   - 파일이 있으면 이어서 추가, 없으면 새 파일 생성
   - 항목 ID는 `CHK001`부터 순차 증가
5. **구조 기준**
   - `.specify/templates/checklist-template.md` 형식을 우선 사용

## 항목 작성 규칙

각 항목은 요구사항 자체를 검증해야 합니다.

- Completeness: 필요한 요구사항이 모두 존재하는가?
- Clarity: 모호한 표현 없이 구체적인가?
- Consistency: 문서 내부/문서 간 충돌이 없는가?
- Measurability: 객관적으로 검증 가능한가?
- Coverage: 주요/대체/예외/복구/비기능 시나리오를 다루는가?

항목 형식 가이드:

- 질문형 문장
- 품질 차원 태그 포함 (`[Completeness]`, `[Clarity]` 등)
- 가능하면 추적 태그 포함 (`[Spec §X.Y]`, `[Gap]`, `[Ambiguity]`, `[Conflict]`, `[Assumption]`)

금지:

- 구현 검증 문장(예: Verify/Test/Confirm + 동작)
- 코드 실행/클릭/렌더링/호출 성공 여부 확인 문장

## 품질 제약

- 항목의 80% 이상은 추적 참조를 포함
- 후보가 40개를 넘으면 위험/영향 기준으로 축약
- 유사 항목은 병합

## Report

아래를 보고합니다.

- 생성된 체크리스트 경로
- 항목 수
- 선택된 포커스/깊이/대상 독자
- 사용자의 must-have 반영 여부

마무리 문구:

`각 /speckit.checklist 실행은 새 체크리스트 문서를 만들거나(또는 동일 도메인 파일에 추가) 독립적으로 관리됩니다.`
